### Overview

* Every `vpc` has a `vpc router`, this router has an interface in every `subnet` within the `vpc` that is reachable from the `subnet` cidr block range + 1
    - Example: if the `subnet` range is `192.168.1.0/24` then the `vpc router interface` ip address is `192.168.1.1`
    - The purpose of the router is to route traffic between `subnets` and in and out of the `vpc`
    - We can control this router by configuring `route tables`

### Route Tables

* A `route table` configures where traffic will get sent. It is a set of routes that the `vpc router` users to forward network traffic
    - Each `rule` in the table is known as a `route`
* The `vpc router` will check traffic leaving the `subnet` and check only the destination IP of the packet and will find a matching `route` within the table by checking the destination column
    *  Its possible for traffic to match multiple `routes` in a table, if thats the case, then the `router` will match to the `rule` that has the larger prefix length (e.g. `/16` is larger than a `/8` so that will get matched)
    - If that traffic matches the `rule` then the `vpc router` will send the traffic to the target specified for that rule
        * All `route tables` will have one `rule/route` by default, which is the `local route` ( 2 `local routes` if IPv6 is enabled)
            - This is done so all traffic originating from the `vpc` that is sending traffic to another resource in the `vpc` will match to the `local` target, allowing routing between resources in a vpc.
        * Any new `subnet` will be default be associated with the default `route table`
        * Multple `subnets` can be associated with the same `route table`
        * A `subnet` can only be associated with one `route table`

### Internet Gateways

* When you create a `subnet` by default it will be private with no access to the internet. In order to make it public we need to make use of an `internet gateway`. These are attached to `vpcs` and are region-resilient (they cover all `AZs` within a region).
* If a `vpc` does not have an `igw` attached to it then all subnets within that `vpc` will be considered private 
* A `vpc` can have one `igw` attached to it and an `igw` can only be attached to one `vpc` at a time
* There's a multi step process for creating a `public subnet`
    1. Create an `igw`
    2. Attach it to a `vpc`
    3. Create a custom `route table`
    4. Create a default `route/rule` routing traffic to the `igw`
        - If there is not a more specific `route` for the traffic (like a `local route` for internal `vpc` traffic) then it will forward to the `igw`
    5. Associate the `subnet` with the custom `route table`

NOTES: 
* Devices created in a `public subnet` and have ` public ip` are unaware of that `public ip`. They are only aware of the `private ip`, these `virtual interfaces` map directly to the `private ip`. So when an external user hits the `public ip` aws translates it to the `private ip`. Multiple public interfaces are all being translated to the `private address`

* Without an `igw` even if you assign a public ip to a device within a `subnet` it will still be unreachable

### NAT Gateways

* `NAT Gateways (ngw)` are meant to solve a complex problem. Your device needs to be able to access the internet but should not be accessible from the internet.
* To use a `ngw` you'll still need to use an `igw`
* The steps are as follows:
    1. Create an `igw` and attach it to a `vpc`
    2. Create a `public subnet` with `igw` associated as default route to `route table`
    3. Deploy `ngw` onto public subnet
    4. Create a `route table` with a default `route` that points to the `ngw`
        - flow from the `private subnet` will be to the `ngw`, which will forward request to the internet through the `igw`

* `ngw` are not region resilient like `igw`, meaning they are specific to an `AZ`, so if that `AZ` goes down then so does your devices' access to the internet
    - For redundancy you'll need multiple `ngw` deployed across `AZs`
        * Each `AZ` should have its own `ngw` and each `AZ` should route its traffic through its own `ngw` ( so u don't incur cross-AZ bandwidth costs)
    - `Regional ngw` are now possible in aws

NOTES:
* You're charged per hour of use of the `ngw` as well as for each gigabyte of data processed
* Supports 5Gbps of bandwidth and auto scales up to 100Gbps
* `Elastic Ips` must be attached to `ngw` so ip where `private rt` routes to, will not change
    - Remember you're creating the `ngw` in the `public sunbet` but adding it as default `route` in the `private rt`

### DNS

* When resources get deployed to a subnet, they'll get a `domain name` (which will usually have the private ip embedded in it) for both the private ip addr
* To resolve these domain names, aws has dns servers that we can use. Which need only be queried. 
    * Globally at `169.254.169.253`
    * Per VPC, as we saw in the [subnets](./2026-06-17-subnets.md) page, the second ip address of a `vpc` routes to the `dns server` (e.g `192.168.10.2`)
        - When you ssh to an ec2, and check the `/etc/resolv.conf` you'll see the nameserver defined there
* By default only `private subnets` get a dns entry, `public subnets` do not 
    - If you would like `public ips` to receive a dns entry, you must click `enableDnsHostnames` when creating a `vpc`
    - If you would like to enable dns resolution for these public dns entries in `public subnets`, you must click `enableDnsSupprt` when creating a `vpc`
        * `dnsSupport` is enabled by default, if disabled you'll have to specify your own nameservers in order to resolve any internet or internal domain names
    * NOTE: you can enable this at anytime, even after a device is created in the `vpc`. At which point aws will automatically assign a dns name to all public ip addresses in that `vpc`

### Elastic Ips

* `Elastic Ips` allow you to reserve a static IPv4 address. You own that ip and now one else can use it. This is because public address given to resources can rotate on reboot. `eips` allow us to hold this ip indefinitely and move it from resource to resource, making migration workflows much easier.
* They are specific to a region, so they can't be moved cross regionally
* You can even bring your own public ip to aws, but typically an `eip` comes from aws' pool of IPv4 addresses
* You can associate an `eip` to an instance at no charge, but associating additional `eips` will incur charge per housr. Reserving an `eip` and not associating with will also incur a charge.
