### SNS

* AWS `Simple Notification Services (SNS)` is a fully managed aws service for both `application-to-application(a2a)` and `application-to-person(a2p)` communication
    - it uses a pub/sub model (publishers and subscribers) to broadcast messages or alerts to multiple subscribing endpoints such as:
        - `lambda`
        - `sqs`
        - `http/s`
        - `email`
        - `sms`
        - `mobile push notifications`

#### Components

![alt text](images/sqs-sns/image.png)

* `Publishers(producers)`: send messages to a logical access point called a `topic`
* `Subscribers(consumers)`: receive messages from `topics` they are subscribed to
    - by sending a single messae to an `sns` topic, you can distribut it to dozens or thousands of parallel subscribers
    - subscribers can define json filter policies so they only receive the specific messages they care about
* `Topics`: logical access point that acts as a communication channel
    ![alt text](images/sqs-sns/image-1.png)
    * `Standard`: offer nearly unlimited throughput
        - can handle unordered or duplicated data
        - can scale
    ![alt text](images/sqs-sns/image-2.png)
    * `FIFO`: designed for strict first-in-first-out sequencing and exactly-once message delivery, capping at 300 messages per second (cannot scale up)
        - strict message ordering and no duplicates 
* `Messages`: the data payload pushed by the publisher to the topic
    - size: each message payload can contain up to 256KB of text data
        * if you need more, you can send up to 2GB of data by storing ref for message in `s3`
    - attributes: publishers can attach metadata attributes (key/value pairs) used for routing or filtering without altering core message body
    - encryption: can be encrypted with `kms`

* NOTE: you can batch requests to save costs
    - each api request can handle up to 10 messages

### SQS

* AWS `Simple Queue Service (SQS)` is a fully managed message queuing service that lets you send, store, and receive messages between software components at any volume, helping you decouple and scale applications without losing messages
    - allows services to communicate asynch
    - if the receiving system goes offline, the messages wait safely in the queue until its ready to process them
        * decoupling messaging from application
        ![alt text](images/sqs-sns/image-3.png)
        ![alt text](images/sqs-sns/image-4.png)
* Follows a producer-consumer model
    - producer sends a message to the queue
    - a consumer processes the message and deletes it when its finished

#### Components

![alt text](images/sqs-sns/image-5.png)
* `Producers`: the application that creates and sends messages to an `sqs queue`
* `Queue`: a temporary buffer distributed across multiple aws servers that stores the messages until they are processed and deleted
    - `Dead Letter Queue (DLQ)`: a separate queue where messages are senf it they repeatedly fail to be processed, protecting your system from bad messages and helps in debugging
    ![alt text](images/sqs-sns/image-6.png)
    - `Standard`: maximum throughput, best effort ordering, and at-least-once message delivery
        * can scale
    ![alt text](images/sqs-sns/image-7.png)
    - `FIFO`: first in and first out quees, guarantees that messages are processed exactly-once and in the exact order they're sent
        * without batching can support up to 300 messages/swith batching supports up to 3000 messages/s (cannot scale)
        * you can enable througput mode to support 9000/per seconds without batching
* `Consumers`: the application that poll the queue using `Receive Message`, process the data, and then delete the message from the queue
* `Message`: the payload of the data
    - size: each message payload can contain up to 256KB of text data
    - attributes: producers can attach metadata attributes (key/value pairs) used for routing or filtering without altering core message body
* `Visibility Timeout`: the period of time where a message retrieved by a consumer becomes invisible to other consumers
    - if the consumer fails to process and delete the message before the timeout, the message becomes visible for another consumer to pickup
* `Message Retention Period`: time `sqs` keeps the message in the queu if it isn't deleted
    - from 1 min to 14 days
* `Message Prioritization`: can set priority for critical messages
